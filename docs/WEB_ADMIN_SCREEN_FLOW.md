# 0.8.0A Web 後台畫面流程

## 使用情境

第一版 Web 後台 MVP 的使用者視角是「廟方實際管理者」。目標不是逐欄位微調，而是先用截圖與操作回饋確認整體資訊架構、管理流程與畫面理解度。

第一版只做可視化畫面與流程驗證，不正式上線，不接正式 LINE Bot，不改正式 Google Sheets，不改 Render，不改 LINE Developers Webhook。

## 資料來源限制

畫面與流程只能使用：

- `中原福德宮_AppSheet_0612_正式啟用前備份_20260616`
- 本機 mock data
- 本機 dev adapter

不得使用正式主檔 `中原福德宮_AppSheet_0612` 作為 Web 後台 MVP 的寫入目標。

## V1 Freeze 與未來 LINE Bot 轉接

`0.8.0A-9` 補充：

- 現有 V1 = Google Sheets + Render FastAPI + LINE Bot 保留但凍結。
- V1 不再新增功能、不再擴充 Google Sheets、不再追隨 V2 prototype 變動。
- 現有 Google Sheets 不再作為 Web 後台 V2 的資料核心或設計限制。
- V2 正式進入 Web 後台架構與開發階段，不再只是 Google Sheets 的前端。
- LINE Bot 不取消；現有 LINE Bot 暫不跟隨 V2 變動。
- 未來 LINE Bot 應待 V2 核心資料模型、權限與 API 邊界穩定後，再轉接 Web 後台 / API / 新資料核心。
- `docs/V1_V2_DATA_MAPPING_PLAN.md` 只作為 legacy reference，不作為 V2 設計限制。
- 詳細決策見 `docs/V1_FREEZE_AND_WEB_ADMIN_TRANSITION_DECISION.md` 與 `docs/DATA_BACKEND_EVOLUTION_STRATEGY.md`。

`0.8.0A-10` 補充正式技術架構規劃：

- 正式 Web 後台前端方向為 Vercel + Next.js / React。
- 後端 API 與未來 LINE Bot webhook 方向為 Render FastAPI。
- V2 正式資料核心方向為 Render PostgreSQL。
- `web_admin_mvp/` 只保留為 UX prototype 與流程參考。
- 未來正式前端建議另開 `web_admin_app/`，但本輪不建立。
- 詳細規劃見 `docs/WEB_ADMIN_TECHNICAL_ARCHITECTURE_ROADMAP.md`、`docs/WEB_ADMIN_APP_DEVELOPMENT_PLAN.md`、`docs/RENDER_API_AND_LINEBOT_FUTURE_INTEGRATION_PLAN.md`。

## 導覽結構

最新 UX 決策：左側選單只保留最上層模組，不列出細項功能。

- 主控台
- 善信管理
- 友宮管理
- 來訪 / 請帖
- 公告 / 活動
- 團隊管理
- 帳務管理
- 管理者設定

目前 `web_admin_mvp/` 已依 V2 架構文件先做成 9 個模組入口：

```text
主控台
善信管理
友宮管理
來訪 / 請帖
公告 / 活動
團隊管理
帳務管理
管理者設定
```

廟務職務與系統權限是團隊管理的下一層內容，不放在左側選單。仍只使用本機測試資料，不接正式 Google Sheets，不修改正式 LINE Bot runtime，不部署。

細項功能不攤在左側選單。一般模組主頁以列表、搜尋、篩選與查看詳情為主；`新增 / 編輯`、聯絡人、職務任期、廟務職務 / 系統權限、收支分類、發布管道設定、查詢維運與資料來源說明不應攤在左側選單或一般主頁快速卡片。

管理者設定集中放置：

- 標籤 / 主檔管理：發布管道、發布狀態、來訪型態、廟務職務、友宮聯絡人職稱、系統權限、帳務分類。
- 權限與角色：系統權限管理、管理者清單、`admin` 可轉移 / 增減 / 停用說明。
- 系統操作紀錄：最近查詢紀錄、查無資料紀錄、補資料建議 / 待補清單、LINE Bot 狀態說明。
- 系統與資料來源：開發資料來源說明、AppSheet 備援說明、正式主檔保護提醒、不影響正式 LINE Bot 說明。

本階段仍不做正式公告發送、自動推播、完整登入管理或正式權限驗證。

## 全站資料管理流程

所有主要資料管理模組共用以下流程：

```text
先瀏覽 / 搜尋 → 查看詳情 → 詳情頁內編輯 / 管理關聯資料
```

分工原則：

- 列表頁負責瀏覽、搜尋、篩選與查看詳情。
- 列表頁每筆資料的主要操作是 `查看詳情`。
- 列表頁不直接提供大量編輯、停用、封存或管理關聯資料。
- 詳情頁才提供編輯、停用 / 封存、管理聯絡人、管理關聯公告、管理權限、管理任期等操作。
- 一般前端不提供刪除；需保留紀錄的資料以停用 / 封存處理。
- 新增 CTA 只保留在明確少數位置，不在列表與卡片中重複出現。
- 快速通道、列表與相關資料區塊不應重複呈現同一批資料。
- 各模組主頁不優先顯示大量快速卡片；若卡片只是重複列表功能，應移除。
- 設定類、權限類、系統操作紀錄、資料來源類集中到管理者設定。

## 模組語意邊界

- 善信管理 = 管人，用來管理善信基本資料、聯絡資料、推播同意與建檔資訊。
- 廟務管理 = 管事，用來管理添油香、光明燈、太歲、發財金、平安龜、還金 / 還願、採購、公文 / 通知與廟務維持事項。
- 帳務管理 = 管錢，用來管理收入、支出、帳務科目、金額、關聯來源、經手人與月報公告草稿。
- 友宮 = 宮廟主檔，用來管理友宮基本資料、聯絡資訊與主檔狀態。
- 來訪 / 請帖 = 友宮互動事件，用來記錄某間友宮在某一天的一次互動。
- 公告 / 活動 = 發布或活動內容，用來整理對外公告、活動資料與發布管道文字。
- 近期廟務動態 = 對人可讀的廟務事件，用於主控台快速掌握最近來訪、公告、活動、善信服務、團隊值勤等異動，不顯示系統 log。
- 系統操作紀錄 = 管理者維運資訊，包含查詢紀錄、查無資料、補資料建議與 LINE Bot 狀態說明，不作為一般使用者主要操作內容。

目前 prototype 應維持：

- 友宮、來訪 / 請帖、公告 / 活動、團隊管理、帳務管理主頁均以列表優先，列表操作以 `查看詳情` 為主。
- 編輯與關聯操作集中到各自詳情頁。
- 關聯資料在主資料詳情頁中只顯示數量、最近一筆或摘要，提供查看 CTA，不在摘要區直接編輯。
- 管理者設定集中處理設定、權限、系統操作紀錄與資料來源。
- 所有資料仍只使用本機測試資料，不接正式 Google Sheets，不影響正式 LINE Bot。

## 廟務管理 / 採購 / 公文通知架構整理

`0.8.0A-8` 只做文件化，暫不實作：

- 「廟務辦理」統一改名為「廟務管理」。
- 善信管理只管人，例如會員編碼、手機、電話、地址、出生日期、是否接受活動 / 廣告推送、key in 人員代號與 key in 日期時間。
- 廟務管理管日常廟務事項，例如添油香、光明燈、太歲、發財金、平安龜、還金 / 還願、採購、公文 / 通知、廟務維持。
- 帳務管理管錢，包含收入、支出、帳務科目、金額、關聯來源、經手人與月報公告草稿。
- 帳務科目 / 費用科目 / 收支科目由帳務管理統一主檔管理，其他模組只能選用既有科目，不應自由輸入。
- 採購是廟務管理子功能；採購管理管品項、數量、用途、店家 / 供應商、經手人與關聯事項，付款與支出流水歸帳務管理。
- 公文 / 通知先歸廟務管理，包含行政單位來文 / 發文、友宮來函 / 回函、內部通知、會議通知、活動協調通知。
- 公文 / 通知是文件紀錄；公告 / 活動是發布內容或活動資料。公文 / 通知可產生公告草稿，但不取代公告 / 活動。
- 首頁未來可增加單純主選項頁面；左側選單未來可自動隱藏 / 點選展開。
- 未來可規劃行事曆呈現來訪、帳務、活動、值勤；友宮詳情未來可規劃地圖呈現。
- 既有專案 `D:\Projects\ExplorePass_user_V1`、`D:\Projects\ExplorePass_V2`、`D:\Projects\Temple_explore` 未來僅可 read-only 參考 UX pattern，不讀取、不修改、不搬程式或資料模型。

## 1. 首頁 / 主控台

用途：讓管理者快速知道目前資料維護狀態。

建議內容：

- 友宮總數
- 近期來訪 / 請帖筆數
- 草稿或待確認公告數
- 近期廟務動態，例如來訪、公告、活動、善信服務或團隊值勤提醒

限制：

- 不做複雜 dashboard。
- 不做即時營運分析。
- 不連正式 LINE Bot 指標。

## 2. 友宮資料列表

用途：查找與維護友宮主資料。

建議功能：

- 模糊搜尋友宮名稱、主神、地址、關係狀態或最近互動
- 顯示主要聯絡人摘要
- 顯示最近互動摘要與日期
- 進入友宮詳情
- 進入新增友宮

主頁需以友宮資料列表為主，不放重複的友宮管理快速卡片。每筆資料操作只保留 `查看詳情`。

列表欄位可先保持精簡：

- 友宮名稱
- 地區
- 主祀神明
- 主要聯絡人
- 最近互動
- 狀態

## 3. 友宮資料詳情

用途：集中呈現單一友宮的主資料、聯絡人與互動紀錄。

建議區塊：

- 基本資料
- 聯絡資訊摘要
- 聯絡人列表
- 近期來訪 / 請帖
- 相關公告
- 內部備註

操作：

- 編輯友宮
- 管理聯絡人
- 新增來訪 / 請帖紀錄
- 查看相關公告
- 新增相關公告
- 停用友宮

關聯資料呈現原則：

- 聯絡人只顯示數量與最近摘要，完整資料進入聯絡人管理或詳情流程。
- 來訪 / 請帖只顯示數量與最近摘要，完整內容進入來訪詳情。
- 相關公告只顯示數量與最近摘要，完整內容進入公告詳情。

## 4. 新增 / 編輯友宮

用途：建立或修正友宮主資料。

建議欄位方向：

- 友宮名稱
- 別名
- 地區 / 地址
- 主祀神明
- 公開摘要
- 內部備註
- 是否公開顯示
- 是否僅限內部查詢

注意：

- 第一版先確認流程與欄位分組。
- 不直接改正式 `shrines` tab。
- Web 後台內部可使用 `temple` 命名，但需透過 adapter 對應現有 `shrines`。

## 5. 友宮聯絡人管理

用途：把友宮主資料與實際聯絡人拆開，避免所有資訊都塞在友宮主檔。

建議功能：

- 新增聯絡人
- 編輯聯絡人
- 標記主要聯絡人
- 停用不再使用的聯絡人
- 從職務下拉選單選擇聯絡人職稱

建議欄位：

- 所屬友宮
- 姓名
- 職務名稱
- 電話
- LINE 或其他聯絡方式
- 備註
- 是否主要聯絡人
- 是否啟用

職務名稱應盡量來自 `role_types`，避免自由輸入造成名稱混亂。

## 6. 來訪 / 請帖紀錄列表

用途：查詢所有來訪、請帖、邀請與廟際互動紀錄。

建議功能：

- 模糊搜尋友宮、日期、來訪型態、回覆狀態或備註
- 顯示紀錄狀態
- 進入新增 / 編輯紀錄

主頁需以來訪 / 請帖紀錄列表為主，不放選擇友宮、型態設定或待補資料快速卡片。每筆資料操作只保留 `查看詳情`。

建議欄位：

- 日期
- 友宮名稱
- 來訪型態
- 主題
- 地點
- 聯絡人

用途：檢視公告草稿、已準備內容與未來可能發布的訊息。

建議功能：

## 10. 團隊管理中的職務 / 權限檢視

用途：讓管理者看懂「廟務職務」與「系統權限」是兩件事。

建議內容：

- 成員列表
- 廟務職務
- 系統權限
- 任期
- 啟用狀態

職務 / 權限檢視是團隊管理的下一層內容，不放在左側選單。頁面需以名單列表為主，不放職稱主檔或系統權限說明快速卡片。每筆資料操作只保留 `查看詳情`。

重要原則：

- 使用者本人目前廟務職稱是總幹事，系統權限是 `admin`。
- `admin` 不應綁定單一個人。
- 未來需可轉移、增減、停用系統權限。

## 11. 善信管理

用途：管理一般善信、發財金、平安龜、還金 / 還願、善信本人可查詢紀錄，以及與帳務收入的關聯。

建議內容：

- 善信資料
- 發財金紀錄
- 平安龜紀錄
- 還金 / 還願紀錄
- 善信查詢紀錄
- 與帳務收入關聯

流程：

```text
善信建檔
→ 登記求發財金 / 平安龜
→ 年度紀錄
→ 還金 / 還願
→ 關聯帳務收入
→ LINE Bot 可查詢本人授權內容
```

權限原則：

- 善信本人只能查詢自己的授權紀錄。
- 團隊成員可協助查詢或登記部分資料，但不等於可管理帳務。
- 系統使用者依權限管理善信紀錄。
- 系統管理者維護分類、年度、權限與資料設定。

## 12. 團隊管理

用途：管理團隊成員、值勤班表、值勤週期、值勤提醒與節日 / 特殊日備註。

建議內容：

- 團隊成員
- 值勤班表
- 值勤週期
- 值勤提醒
- 節日 / 特殊日備註
- 與活動 / 來訪 / 公告關聯

重要原則：

- 團隊成員可以不是系統使用者。
- 值勤班表不等於職務任期。
- 值勤提醒可與 LINE 通知或公告有關，但本階段不做正式推播。

## 13. 帳務管理

用途：檢視未來帳務流水紀錄管理流程，不做完整會計、報稅、審計或銀行帳務功能。

命名原則：

- 未來正式模組名稱使用「帳務管理」。
- 功能描述使用「帳務流水紀錄」、「收支流水帳」、「月報公告草稿」。
- 避免使用容易被誤解為正式會計平台的命名。

建議內容：

- 帳務流水紀錄列表
- 模糊搜尋日期、收支、分類、項目、關聯來源、經手人或狀態
- 月報公告草稿

操作原則：

- 主頁只保留 `新增帳務流水` 作為主要 CTA。
- 帳務流水紀錄列表每筆只保留 `查看詳情`。
- 編輯帳務流水紀錄、儲存草稿、標記作廢、補備註與類別檢視放在帳務詳情頁。
- 收支分類與資料保護提醒集中在管理者設定。
- 帳務測試資料不得包含銀行帳號、帳戶、真實收據號碼、真實個資或敏感資訊。
- 帳務流水紀錄未來需支援關聯善信、發財金、平安龜、還金 / 還願、友宮來訪、活動或公告。
- 每月 LINE 群組公告是必要需求，但本階段只規劃月報公告草稿，不做正式推播。
- 現場表單需有顯性 `經手人 / 現場值班` 欄位，可由當日值勤班表預設，不應只依賴登入 session。
- 金額輸入未來需清洗全形數字、千分位逗號、中文單位、空白與非數字符號。
- 作廢不應直接刪除原始流水，也不以原金額乘以 -1 作為主要策略；應保留原紀錄、標示作廢，必要時另建沖銷紀錄。

月度公告對象至少包含：

- 團隊成員
- 管理委員會成員
- 系統管理者

權限原則：

- 團隊成員可接收必要的月度公告或摘要。
- 管理委員會成員可查看較完整流水帳摘要。
- 系統管理者可查看完整紀錄與設定公告範圍。
- 善信本人只可查詢與自己相關的授權紀錄，不可查看完整內部帳務流水。
- 公開公告與內部公告內容應分開。
