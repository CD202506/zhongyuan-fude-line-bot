# 0.8.0A Web 後台畫面流程

## 使用情境

第一版 Web 後台 MVP 的使用者視角是「廟方實際管理者」。目標不是逐欄位微調，而是先用截圖與操作回饋確認整體資訊架構、管理流程與畫面理解度。

第一版只做可視化畫面與流程驗證，不正式上線，不接正式 LINE Bot，不改正式 Google Sheets，不改 Render，不改 LINE Developers Webhook。

## 資料來源限制

畫面與流程只能使用：

- `中原福德宮_AppSheet_0612_正式啟用前備份_20260616`
- 本機 mock data
- 本機 dev adapter

不得使用正式主檔 `中原福德宮_AppSheet_0612` 作為 Web 後台 MVP 的寫入目標。

## 導覽結構

最新 UX 決策：左側選單只保留最上層模組，不列出細項功能。

- 主控台
- 友宮管理
- 來訪 / 請帖
- 公告 / 活動
- 成員 / 職務
- 財務管理
- 管理者設定

細項功能放在各模組頁面內，以 CTA、卡片、分頁或區塊呈現。`新增 / 編輯`、聯絡人、職務任期、收支分類、發布管道設定、查詢維運與資料來源說明不應攤在左側選單。

管理者設定集中放置：

- 基礎設定：來訪型態設定、發布管道設定、職稱主檔、收支分類。
- 權限與角色：系統權限管理、管理者清單、職務任期設定、`admin` 可轉移 / 增減 / 停用說明。
- LINE / 查詢維運：最近查詢紀錄、查無資料紀錄、補資料建議 / 待補清單、LINE Bot 狀態說明。
- 系統與資料來源：開發資料來源說明、AppSheet 備援說明、正式主檔保護提醒、不影響正式 LINE Bot 說明。

本階段仍不做正式公告發送、自動推播、完整登入管理或正式權限驗證。

## 全站資料管理流程

所有主要資料管理模組共用以下流程：

```text
先瀏覽 / 搜尋 → 查看詳情 → 詳情頁內編輯 / 管理關聯資料
```

分工原則：

- 列表頁負責瀏覽、搜尋、篩選與查看詳情。
- 列表頁每筆資料的主要操作是 `查看詳情`。
- 列表頁不直接提供大量編輯、停用、刪除或管理關聯資料。
- 詳情頁才提供編輯、停用、刪除、管理聯絡人、管理關聯公告、管理權限、管理任期等操作。
- 新增 CTA 只保留在明確少數位置，不在列表與卡片中重複出現。
- 快速通道、列表與相關資料區塊不應重複呈現同一批資料。

目前 prototype 應維持：

- 友宮、來訪 / 請帖、公告 / 活動、成員 / 職務、財務管理列表頁均以 `查看詳情` 為主。
- 編輯與關聯操作集中到各自詳情頁。
- 管理者設定集中處理設定、權限、LINE 維運與資料來源。
- 所有資料仍只使用 mock/dev data，不接正式 Google Sheets，不影響正式 LINE Bot。

## 1. 首頁 / 主控台

用途：讓管理者快速知道目前資料維護狀態。

建議內容：

- 友宮總數
- 近期來訪 / 請帖筆數
- 草稿或待確認公告數
- 最近更新的友宮
- 最近新增的來訪 / 請帖

限制：

- 不做複雜 dashboard。
- 不做即時營運分析。
- 不連正式 LINE Bot 指標。

## 2. 友宮資料列表

用途：查找與維護友宮主資料。

建議功能：

- 依名稱搜尋
- 依地區或分類篩選
- 顯示主要聯絡人摘要
- 顯示最近一次來訪 / 請帖日期
- 進入友宮詳情
- 進入新增友宮

列表欄位可先保持精簡：

- 友宮名稱
- 地區
- 主祀神明
- 主要聯絡人
- 最近互動
- 狀態

## 3. 友宮資料詳情

用途：集中呈現單一友宮的主資料、聯絡人與互動紀錄。

建議區塊：

- 基本資料
- 聯絡資訊摘要
- 聯絡人列表
- 近期來訪 / 請帖
- 相關公告
- 內部備註

操作：

- 編輯友宮
- 管理聯絡人
- 新增來訪 / 請帖紀錄
- 查看相關公告

## 4. 新增 / 編輯友宮

用途：建立或修正友宮主資料。

建議欄位方向：

- 友宮名稱
- 別名
- 地區 / 地址
- 主祀神明
- 公開摘要
- 內部備註
- 是否公開顯示
- 是否僅限內部查詢

注意：

- 第一版先確認流程與欄位分組。
- 不直接改正式 `shrines` tab。
- Web 後台內部可使用 `temple` 命名，但需透過 adapter 對應現有 `shrines`。

## 5. 友宮聯絡人管理

用途：把友宮主資料與實際聯絡人拆開，避免所有資訊都塞在友宮主檔。

建議功能：

- 新增聯絡人
- 編輯聯絡人
- 標記主要聯絡人
- 停用不再使用的聯絡人
- 從職務下拉選單選擇聯絡人職稱

建議欄位：

- 所屬友宮
- 姓名
- 職務名稱
- 電話
- LINE 或其他聯絡方式
- 備註
- 是否主要聯絡人
- 是否啟用

職務名稱應盡量來自 `role_types`，避免自由輸入造成名稱混亂。

## 6. 來訪 / 請帖紀錄列表

用途：查詢所有來訪、請帖、邀請與廟際互動紀錄。

建議功能：

- 依友宮篩選
- 依日期範圍篩選
- 依來訪型態篩選
- 顯示紀錄狀態
- 進入新增 / 編輯紀錄

建議欄位：

- 日期
- 友宮名稱
- 來訪型態
- 主題
- 地點
- 聯絡人
- 狀態

來訪次數不手動填，未來應由同一 `temple_id` 的 `temple_visits` 紀錄數自動計算。

## 7. 新增 / 編輯來訪紀錄

用途：建立來訪、請帖、邀請、祝壽、遶境等互動紀錄。

建議欄位：

- 友宮
- 日期
- 時間
- 地點
- 主題
- 來訪型態
- 方向
- 相關聯絡人
- 內容摘要
- 內部備註
- 是否需後續公告

來訪型態需支援多選，例如：

- 參訪
- 用餐
- 進香
- 邀請
- 請帖
- 祝壽
- 遶境
- 會香
- 聯誼
- 其他

## 8. 公告列表

用途：檢視公告草稿、已準備內容與未來可能發布的訊息。

建議功能：

- 依狀態篩選
- 依日期排序
- 顯示是否關聯來訪紀錄
- 進入新增 / 編輯公告

限制：

- 第一版不做正式公告發送。
- 第一版不做自動推播。
- 第一版不接 LINE 主動推播。

## 9. 新增 / 編輯公告

用途：整理未來可能由 LINE 或其他管道發布的公告內容。

建議欄位：

- 標題
- 內容
- 狀態
- 活動日期
- 地點
- 相關來訪紀錄
- 相關活動
- 內部備註

未來 `announcements` 需能關聯 `temple_visits` 或 `events`，可採 `announcement_links` 或 `announcements.related_visit_id` 方向討論。

## 10. 成員 / 職務 / 權限檢視

用途：讓管理者看懂「廟務職務」與「系統權限」是兩件事。

建議內容：

- 成員列表
- 廟務職務
- 系統權限
- 任期
- 啟用狀態

重要原則：

- 使用者本人目前廟務職稱是總幹事，系統權限是 `admin`。
- `admin` 不應綁定單一個人。
- 未來需可轉移、增減、停用系統權限。

## 截圖回饋方式

第一版回饋建議以畫面截圖或簡短操作流程為主：

- 哪個畫面分類不直覺。
- 哪個流程不像廟方實際工作方式。
- 哪些欄位分組應調整。
- 哪些功能應延後，不應干擾第一版。

不建議在第一輪逐欄位微調，避免過早把 V1 欄位限制帶入 V2 後台設計。
